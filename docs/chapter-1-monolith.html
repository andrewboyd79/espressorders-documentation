Chapter 1

The Monolith

A Django application that was never meant to become a microservices platform — until it did.

Django Aurora PostgreSQL S3 AWS ECS ALB

The problem

EspressOrders began as a project for a separate certification — a standard Django application with a PostgreSQL database, S3 for static and media assets, and a session-based authentication system. It was functional, reasonably structured, and entirely typical of a monolithic web application.

It was also not designed to be decomposed. The codebase had the normal accumulation of assumptions that any real project develops: tightly coupled views and models, session-based auth that would not translate easily to stateless microservices, and no API layer because none was needed at the time.

Rather than start from a clean demo project, I chose to work with this existing codebase. That constraint was the point — real DevOps and cloud engineering work happens on imperfect codebases, not greenfield ones.

What was built in this chapter

The monolith was containerised and deployed to AWS ECS behind an Application Load Balancer, with Aurora PostgreSQL as the database and S3 for static file serving. The infrastructure was defined in CloudFormation, following the same repeatable pattern used by every subsequent service.

The constraint that shaped everything

Django's session-based authentication stores session tokens in a cookie. Mobile clients and other services cannot easily consume cookie-based auth. This constraint — inherited from the original codebase — drove the later decision to build a dedicated JWT identity service rather than modify the monolith's auth system.

That is what makes this chapter interesting from an engineering perspective: the monolith was not cleaned up or idealised. It was deployed as-is, and every constraint it introduced was addressed downstream through deliberate architectural decisions.

Key trade-offs

  • The monolith continued running throughout the decomposition — the Strangler Fig pattern required it
  • Session-based auth was left in place rather than rewritten — a deliberate deferral, not an oversight
  • No API layer was added at this stage — that came with espressorders-ms in Chapter 2
