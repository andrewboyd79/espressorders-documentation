Chapters 2–4

Decomposition

Extracting four microservices from a running monolith — without taking the system down.

FastAPI Strangler Fig AWS ECS DynamoDB SNS / SQS Stripe Aurora PostgreSQL

The problem

A monolith that does everything is easy to deploy and hard to scale. As EspressOrders grew, the coupling between catalogue browsing, cart management, order processing, and notifications made independent deployment impossible — a change to the order flow required redeploying the entire application.

The question was not whether to decompose, but how. A big-bang rewrite would mean months of parallel development with no user-facing changes and high risk of regression. There had to be a better way.

The decision: Strangler Fig

The Strangler Fig pattern extracts functionality incrementally behind a facade — the original monolith keeps running, the facade routes traffic to whichever implementation (monolith or new service) is ready. At no point is the system unavailable, and at every step the change is reversible.

espressorders-ms became that facade — a clone of the original Django app, progressively rewritten to consume microservice APIs instead of the monolith's internal services. The monolith itself was not modified.

The services extracted

  • Catalogue — read-only FastAPI service, Aurora-backed, consumed by the facade for menu browsing
  • Cart — FastAPI service with DynamoDB for session-scoped cart state; stateless, horizontally scalable
  • Order — FastAPI service handling Stripe payment processing and order creation; publishes to SNS on completion
  • Notification — FastAPI service consuming from SQS, sending order confirmation emails via SES

The Saga pattern

Order processing spans multiple services: payment (Stripe), order creation, and notification. A distributed transaction that fails halfway must be recoverable. The Saga pattern — implemented via SNS/SQS — ensures each step either completes or publishes a compensating event. The Order service publishes to SNS on success; the Notification service consumes from SQS asynchronously. No two-phase commit, no tight coupling.

Key trade-offs

  • The monolith remained deployed throughout — operational overhead, but zero migration risk
  • DynamoDB for cart state was chosen over Aurora for horizontal scalability and session-scoped access patterns
  • Asynchronous notification via SNS/SQS means emails may arrive seconds after order confirmation — acceptable for this use case

The full decomposition decision — options considered, criteria applied, and trade-offs accepted — is documented in ADR-001 in the private repository.
