Chapter 4 · Mobile

Mobile Client

A React Native app built against an existing API — the point where every earlier design decision either pays off or costs you.

The problem

A coffee ordering platform without a mobile client is incomplete. The web application works, but in 2026 consumers expect to order from their phone — and the portfolio needed to demonstrate cross-platform thinking alongside the backend and infrastructure work.

The question was whether the backend, built without a mobile client in mind, would support one without modification. API-first design is only proven when a second consumer arrives.

The result: zero backend changes

The mobile client was built against the existing APIs without modification to any backend service. Every design decision made in earlier chapters — stateless JWT authentication, RESTful service boundaries, Stripe payment intent API — mapped cleanly to mobile patterns. The backend required no changes.

This was not luck. It was the consequence of building services with clean interfaces rather than interface-per-consumer — a discipline that is easy to skip when there is only one consumer.

Technology decisions

Expo over bare React Native — managed workflow removes native build toolchain complexity for a portfolio project. EAS Build handles iOS and Android compilation on Expo's infrastructure.

JWT in SecureStore — tokens are stored in the device's secure enclave, not in AsyncStorage. The same security posture as the browser's httpOnly cookie, implemented for a native context.

Preview profile (iOS simulator) over App Store — a full App Store distribution build requires an Apple Developer Program account ($99/year). The preview profile produces an iOS simulator build via EAS without requiring certificates or a paid account. The CI/CD pipeline is fully demonstrated; the final distribution step is a known, documented prerequisite.

CI/CD pipeline

The mobile app uses the same CodeBuild + GitHub webhook pattern as every other service. Two jobs: a CI job on the dev branch (TypeScript type check, Jest tests) and a release job on the prod branch (CI + EAS simulator build). The EAS token and API credentials are pulled from SSM Parameter Store at build time — stored in the Infra account where CodeBuild runs.

Key trade-offs

  • Expo managed workflow over bare React Native — less control, significantly less complexity for a solo portfolio project
  • Preview (simulator) build over production build — documented gap, not a hidden limitation
  • Polling for order status (10-second interval) over WebSocket — simpler to implement, sufficient for the demo use case
Architectural Decision Record · ADR-004 Accepted

Use Expo managed workflow with preview build profile

Context

A React Native mobile client is needed for the portfolio. Full App Store distribution requires a paid Apple Developer account ($99/year). The goal is to demonstrate the full CI/CD pipeline, not distribute to the App Store.

Decision

Use Expo managed workflow with EAS Build. Use the preview profile to produce an iOS simulator build — demonstrates the full pipeline without requiring paid certificates.

Consequences

Full CI/CD pipeline demonstrated. Build runs on Expo's infrastructure via CodeBuild. Known gap: simulator build only, not device-installable without Apple Developer account. This is documented, not hidden.
