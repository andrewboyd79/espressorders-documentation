About this project

A real codebase, evolved deliberately.

EspressOrders started as a Code Institute Diploma project — a standard Django monolith with a PostgreSQL database, S3 for assets, and session-based authentication. Rather than leaving it there, it became the subject of an extended engineering exercise:

  • decomposing it into microservices
  • adding operational maturity
  • building a React Native mobile client against the resulting APIs

Each stage introduced real constraints and required real decisions. This documentation captures those decisions — what was built, why it was built that way, and what each choice made easier or harder downstream.

"Every architectural decision here was made against a real codebase with real constraints — not a greenfield project designed to make the pattern look easy."

What this documents

Architecture, infrastructure, and the decisions behind them.

Each chapter documents a stage of the project — what was built, what was decided, and why.

The Architecture

Microservices via Strangler Fig

Four FastAPI services extracted from a Django monolith — Catalogue, Cart, Order, and Notification — with event-driven coordination via SNS/SQS.

The Infrastructure

AWS ECS, CloudFormation, CodeBuild

Every service deployed via Infrastructure as Code, with a consistent two-repo pattern and centralised CI/CD pipeline across all eight services.

The Decisions

Architectural Decision Records

Each significant choice is captured as an ADR — what options were considered, what was decided, and what the consequences were.
