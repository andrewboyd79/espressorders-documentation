Chapter 1 · Monolith

Establishing the Platform Foundation

How working with existing systems builds deeper engineering insight than greenfield projects.

The problem

EspressOrders began as my final project for a Diploma in Full Stack Web Development at the Code Institute — a standard Django application with a PostgreSQL database, S3 for static and media assets, and a session-based authentication system. It was functional, reasonably structured, and entirely typical of a monolithic web application.

It was also not designed to be decomposed. The codebase had the normal accumulation of assumptions that any real project develops: tightly coupled views and models, session-based auth that would not translate easily to stateless microservices, and no API layer because none was needed at the time.

Rather than start from a clean demo project, I chose to work with this existing codebase. That constraint was the point — real DevOps and cloud engineering work happens on imperfect codebases, not greenfield ones.

What was built in this chapter

The monolith was containerised and deployed to AWS ECS behind an Application Load Balancer, with Aurora PostgreSQL as the database and S3 for static file serving. The infrastructure was defined in CloudFormation, following the same repeatable pattern used by every subsequent service.

The constraint that shaped everything

Django's session-based authentication stores session tokens in a cookie. Mobile clients and other services cannot easily consume cookie-based auth. This constraint — inherited from the original codebase — drove the later decision to build a dedicated JWT identity service rather than modify the monolith's auth system.

That is what makes this chapter interesting from an engineering perspective: the monolith was not cleaned up or idealised. It was deployed as-is, and every constraint it introduced was addressed downstream through deliberate architectural decisions.

Key trade-offs

Strangler Fig

Monolith kept running throughout

The Strangler Fig pattern required the monolith to remain deployed throughout the decomposition — operational overhead, but zero migration risk at each step.

Auth deferral

Architectural Decision Record · ADR-001 Accepted

Work with the existing monolith rather than a greenfield project

Context

A portfolio project needs a realistic starting point. Greenfield projects are clean but artificial — real DevOps and platform engineering work happens on codebases with accumulated assumptions, constraints, and technical decisions already baked in.

Decision
  • Use the existing Code Institute Diploma Django application as-is.
  • Deploy it to AWS ECS without refactoring the codebase.
  • Accept all inherited constraints — session-based auth, no API layer, tightly coupled views — and address them downstream.
Consequences

Every constraint that followed — the JWT identity service, the Strangler Fig migration, the lack of an API layer at this stage — was real and inherited, not invented. The architectural decisions in later chapters are responses to genuine problems, not contrived ones.
