Platform Automation and Environment Orchestration

Building self-service platforms that enable teams to move fast while maintaining governance and compliance.

Platform Consistency as Force Multiplier

Every service in this platform follows the same infrastructure pattern. Not because it was planned upfront, but because applying the same approach eight times made the reasoning obvious: one decision, made well, multiplied across every repo.

This approach delivers three platform advantages:

  1. Reduced cognitive load — teams can focus on business value rather than relearning infrastructure patterns
  2. Accelerated delivery — new capabilities can be added by copying proven patterns rather than redesigning from scratch
  3. Enhanced AI assistance — predictable structures enable AI tools to provide meaningful context-aware suggestions

"Decide the pattern once. Every repo follows it. Only diverge when you have a genuine reason."

The repo pattern

Every microservice follows a two-repo structure: an application repo (the service code) and a service-config repo (the CloudFormation infrastructure for that service). All eight service-config repos are structurally identical:

Both repos use the same branch model: dev for active development, prod for stable releases. The same CodeBuild pipeline pattern triggers on push to each branch.

Shared infrastructure

Beneath the service layer, shared infrastructure is managed in dedicated repos — also following consistent patterns:

ecs-infrastructure

ECS cluster and load balancer

ECS cluster, Application Load Balancer, Route 53 A record. Deployed once; all services attach to it.

vpc-infrastructure

Network foundation

VPC, subnets, and security groups. The network layer everything else runs on.

task-definition

ECS task definitions

Task definitions for all services. The primary pipeline trigger: push here to kick off a rolling deployment across the platform.

parameters

SSM parameter management

All SSM parameters managed as infrastructure-as-code. No ad-hoc CLI parameter creation; every secret and config value is tracked in version control.

buildspecs

Centralised CodeBuild YAML

Build specs for all services in one repo. Synced to S3 on push to dev; CodeBuild reads from S3 at build time.

deployment-cli

Pipeline and ECR tooling

CloudFormation templates for CodeBuild projects and ECR repository policies. The tooling that creates the pipelines.

CI/CD pipeline pattern

Every service uses the same pipeline pattern: a CodeBuild project with a GitHub webhook trigger, parameterised by branch, deployed via CloudFormation from deployment-cli. The buildspec lives in the centralised buildspecs repo rather than the application repo — a single place to update build configuration across the entire platform.

One-command environment management

Running nine ECS services continuously is expensive. For a portfolio project, the environment needs to be spun up for demonstrations and shut down afterwards. The espressorders-spinup-automation repo provides this: a Step Function that orchestrates the spin-up or spin-down of the entire environment in strict dependency order, triggered by a single CLI command.

This is the same pattern a platform engineering team would put behind a "Start Environment" button in an internal developer portal — complex orchestration abstracted to a single, repeatable action.

The dependency order matters: the ECS cluster must exist before services can attach to it; the ALB must exist before listener rules can be created; Aurora must be running before services that depend on it start. The Step Function enforces this order automatically, eliminating the manual coordination that would otherwise be required.

Why this matters for production

The patterns here are not academic. The two-repo structure, centralised buildspecs, infrastructure-as-code parameters, and automated environment orchestration are all patterns that translate directly to enterprise platform engineering:

  1. A new team service can be onboarded in hours by copying the existing pattern
  2. Environment promotion (dev → prod) is a single parameter change, not a process document
  3. The platform is auditable — every parameter, every pipeline, every infrastructure change is tracked in version control
  4. It can be torn down and rebuilt from scratch — which means it can be ported to another project or organisation
Architectural Decision Record · ADR-010 Accepted

Use a two-repo pattern for every microservice

Context

Each service needs application code and infrastructure code. Keeping them together couples deployment tooling to application changes. Separating them allows infrastructure to be updated independently, and makes the pattern explicit rather than implied.

Decision
  • Every service gets two repos: one for application code, one for CloudFormation infrastructure.
  • Both repos use the same branch model: dev for active development, prod for stable releases.
  • Build specs live in a centralised buildspecs repo, not in individual service repos.
Consequences

Onboarding a new service means copying a known pattern rather than designing from scratch. Infrastructure changes are auditable in version control independent of application changes. The centralised buildspec pattern means a single update propagates to all eight pipelines without touching each service repo.
